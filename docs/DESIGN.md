# Paw:Paw 디자인 브리프 (v2 · 실제 디자이너 산출물 기준)

> 이 문서는 디자이너가 Figma Make로 만든 대시보드/AI챗봇/마이펫이지 3개 화면의
> `src/imports/guidelines.md`(세 프로젝트 모두 동일 문서 확인됨)와 실제 구현 코드의
> hex 값을 대조 검증해서 반영한 **실제 디자인 시스템**입니다. v1(추정 팔레트)을 대체합니다.
>
> **레퍼런스 현황**: 대시보드·AI챗봇·마이펫이지는 `reference/figma-make/`에 실제 구현 코드가 있습니다.
> **메인페이지(간소화 대시보드)와 건강 레포트는 참고할 구현이 없으므로, 아래 토큰/원칙을 기준으로
> Claude Code가 처음부터 새로 디자인해야 합니다.** 이 두 화면을 작업할 때는 레퍼런스 3개 화면의
> 톤(카드 스타일, 컬러 사용 원칙, NAV바 등)을 참고해서 이질감 없이 만드는 것이 특히 중요합니다.

## 브랜드 컨셉 (원문 그대로)
- 서비스명: Paw:Paw (Paw + Paw = 강아지와 고양이 두 마리가 함께하는 돌봄)
- 슬로건: "우리 아이의 하루를, AI가 함께 돌봐요"
- 톤앤매너: 병원처럼 차갑지 않고 다정하고 따뜻한 돌봄의 느낌. 부드러운 3D 클레이/펠트 질감 캐릭터 일러스트 + 라운드 컴포넌트로 친근함 강조
- 로고 모티프: 발바닥(paw) 모양이 "P"자와 결합된 워드마크. 원형 그라데이션 링(민트→퍼플) 안에 심전도 파형(건강) + 막대그래프(데이터 트래킹) 아이콘

## 타이포그래피
- 실제 구현 기준 폰트: **Noto Sans KR** (300~800 전 웨이트 로드됨). 가이드라인 원문은 "Pretendard 또는 유사 라운드 고딕"이라고 열어뒀는데, 레퍼런스 3개 화면이 이미 Noto Sans KR로 통일되어 있으니 **일관성을 위해 Noto Sans KR을 그대로 채택**합니다.
- 워드마크 "Paw:Paw"는 앞부분(Paw) `ink`, 뒷부분(:Paw) `primary-purple`로 투톤 처리 — 신규 화면에서도 로고는 항상 이 규칙 유지
- 위계: 페이지 타이틀(Bold, 큼) → 섹션 타이틀(Semibold, 중간) → 본문/라벨(Regular) → 보조 텍스트(연한 톤, 작게)
- 성능 팁: 6단계 웨이트(300~800)를 다 로드하면 무거워짐 — 실사용 웨이트는 400/500/600/700 정도로 subset 지정 권장

## 레이아웃 & 컴포넌트 공통 규칙
- **NAV바(전 페이지 공통)**: 좌측 로고+슬로건, 중앙 4개 메뉴(현재 페이지는 민트로 활성 표시), 우측 반려인 이름+프로필(클릭 시 반려동물 리스트 드롭다운)
- **카드**: 모서리는 큼직한 라운드(각진 느낌 지양), 그림자는 은은하게. 카드 배경색으로 상태 구분(민트=완료, 옐로우=일정, 화이트/그레이=기본정보)
- **버튼**: 라운드형. Primary는 민트 또는 보라 배경+화이트 텍스트, Secondary는 아웃라인/텍스트 링크
- **숫자 표기(n/m 형식)**: 미완료(n<m)는 coral, 완료(n=m)는 ink로 표기 — `docs/SPEC.md`의 복약 완료율 규칙과 정확히 일치함, 색상만 이 토큰으로 통일
- **캐릭터 일러스트**: 손그림풍/클레이 질감 유지, 말풍선으로 감성적 상태 표현("배고파여" 등) — 증상이 기록되면 말풍선/표정이 데이터에 따라 바뀌는 것을 시그니처 요소로 유지 (가이드라인 원문에서도 이 표현 방식을 명시적으로 허용)
- **AI 챗봇 UI**: 답변에는 항상 출처 표기, 병명 확정형 금지 → "~일지도 몰라요" 톤이 말풍선 스타일에도 시각적으로 드러나야 함 (`docs/features/chatbot-consult.md`와 동일 규칙)

## 메인페이지·건강 레포트·인트로 페이지 신규 디자인 시 참고할 것
레퍼런스가 없는 이 화면들은 아래를 그대로 지키면 다른 화면과 이질감이 없습니다.
- 메인페이지(간소화 대시보드): NAV바 규칙 동일 적용, D-Day 카드는 amber, 복약 캘린더는 완료=mint/미완료=coral. 중앙 마스코트 애니메이션 자리, 말풍선은 알림 우선순위 로직 반영 (`docs/features/dashboard.md` 참고)
- 건강 레포트: 체중 그래프는 `docs/SPEC.md`의 보간 규칙(미입력일은 전날 값 유지)이 시각적으로도 끊김 없이 보이게, AI 의심사항 텍스트에는 반드시 출처 표기 + purple 포인트로 "AI가 분석한 부분"임을 구분
- 인트로 페이지(로그인 전 첫 화면): 히어로 텍스트 + 3개 특징 카드 + 로그인 카드(카카오 버튼만) 구조. `docs/features/intro.md` 참고 — 레이아웃 구조는 참고 자료를 따르되 색상은 반드시 위 토큰(민트/퍼플/아이보리)만 사용, 다른 톤(예: 블루 계열)을 가져오지 않는다

## 성능/무게 가이드라인 (반드시 지킬 것)
1. 클레이/펠트 질감 캐릭터에 Lottie를 쓴다면 마스코트 한 곳에만, 상태별 3~4개로 제한하고 지연 로드
2. 폰트 웨이트는 실사용 4개(400/500/600/700)로 제한, 6단계 전부 로드하지 않기
3. 마스코트 외 모션은 CSS transition으로, 별도 스크롤 애니메이션 라이브러리 도입하지 않기
4. 색상은 위 토큰만 사용 — 레퍼런스 코드처럼 `style={{ color: '#...' }}` 인라인 하드코딩 금지, 반드시 `tailwind.config`에 토큰으로 등록 (`client/CLAUDE.md` 규칙과 동일)

## 성능/무게 가이드라인 (반드시 지킬 것)
포트폴리오용이라도 로딩이 느리면 시연에서 감점 요인이 된다. 아래를 지킨다.

1. **Lottie는 마스코트 한 곳에만.** 상태별로 3~4개 애니메이션(평온/걱정/배고픔/기쁨) 정도로 제한하고, 다른 UI 요소(버튼, 카드)에는 Lottie를 쓰지 않는다.
2. **Lottie 라이브러리와 JSON은 지연 로드.** 메인/대시보드처럼 마스코트가 실제로 보이는 페이지에서만 동적으로 불러오고, 다른 페이지(리포트, 마이펫이지)에서는 로드하지 않는다.
3. 마스코트 외의 모션은 CSS transition(호버·탭 피드백 수준)으로 처리하고, 별도의 스크롤 애니메이션 라이브러리는 도입하지 않는다.
4. 그래프(체중 변화)는 가벼운 SVG 기반 라이브러리 하나만 선정해서 통일 사용 — 차트 라이브러리를 기능별로 다르게 쓰지 않는다.
5. 웹폰트는 위에 정한 2개 서체, 서체당 2~3 굵기로 제한하고 필요한 subset만 로드한다.
