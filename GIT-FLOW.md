# 🚀 [Paw-Paw] 통합 개발 및 형상관리 가이드라인

> 💡 **본 문서는 팀의 효율적인 협업과 코드 품질 유지를 위한 형상관리 및 개발 프로세스 규칙을 정의합니다.**
> 
> 
> 모든 개발자는 개발 시작 전 본 문서를 숙지하고, 정해진 규칙에 따라 커밋, 이슈, PR을 관리해야 합니다.
> 

## 🚨 우리 팀의 Git 협업 약속

- `main` 브랜치와 `dev` 브랜치로의 직접 `push`는 절대 금지합니다.
- 기능 개발은 반드시 각자의 작업 브랜치(`feat/...`)에서 진행합니다.
- 작업이 끝나면 원격 저장소에 브랜치를 push하고, GitHub에서 PR을 작성해 상대방의 코드 리뷰 및 승인을 얻은 뒤 `dev` 브랜치에 머지합니다.
- 한별: HB
- 학래: HR

## 📌 1. Git Branch 전략 (Git Flow 기반)

우리 프로젝트는 `feat ➡️ dev ➡️ main`으로 이어지는 안정적인 브랜치 전략을 사용합니다.

### 📂 브랜치 구조 및 역할

- **`main`** : 최종 사용자에게 서비스되는 **실제 배포용 브랜치**입니다. 가장 안정적인 최적의 상태를 유지합니다.
- **`dev`** : 각자 개발한 기능들이 모여서 **코드 병합 및 통합 테스트를 진행하는 중심 브랜치**입니다.
- **`feat/...`** : 새로운 기능 개발, 버그 수정 등 **실제 작업**이 이루어지는 개발 브랜치입니다.

### 🏷️ 브랜치 명명 규칙 (Branch Naming Rule)

원격 저장소의 브랜치 목록을 직관적으로 관리하기 위해 브랜치는 `작업타입/담당자-#이슈번호-간략한설명` 형태로 작성합니다.

| **작업 타입** | **설명** | **예시** |
| --- | --- | --- |
| `feat` | 새로운 기능 추가 | `feat/john-#3-kakao-login` |
| `fix` | 버그 수정 | `fix/seungmin-#12-auth-token-error` |
| `docs` | 문서 작성 및 수정 | `docs/john-#5-readme-update` |
| `refactor` | 코드 리팩토링 (기능 변경 없음) | `refactor/seungmin-#8-axios-interceptor` |

## 📌 2. GitHub Issue 작성 규칙

모든 작업(Feat, Bug Fix 등)은 **이슈를 생성하는 것부터 시작**합니다. 작업 내역을 투명하게 추적하고 유기적으로 연결하기 위함입니다.

> ℹ️ **이슈 템플릿 (Issue Template)**
> 
> 
> 이슈를 생성할 때 아래 내용을 복사해서 작성해 주세요.
> 

Markdown

```
### 📝 작업 개요
- 작업할 내용을 한 줄로 요약해 주세요.

### 🛠️ 세부 작업 내용
- [ ] 첫 번째 하위 작업
- [ ] 두 번째 하위 작업

### 🔗 참고 사항 (선택)
- 관련 링크나 참고할 이미지 등을 첨부해 주세요.
```

## 📌 3. Commit Message 규칙 (Conventional Commits)

커밋 메시지만 보고도 어떤 파일에서 무슨 작업이 일어났는지 직관적으로 알 수 있도록 규칙을 통일합니다. 깃허브 내부 연동을 위해 **이슈 번호 앞에는 반드시 `#`을 붙입니다.**

### ✉️ 커밋 메시지 기본 구조

Plaintext

```
<타입>: #<이슈번호> - <설명>
```

### 🏷️ Commit Type 명세

- **`feat`** : 새로운 기능 추가
- **`fix`** : 버그 수정
- **`docs`** : 문서 수정 (Markdown, 텍스트 파일 등)
- **`style`** : 스타일링, 포맷터 적용 (코드 로직 변경 없는 세미콜론 누락, 인덴트 수정 등)
- **`refactor`** : 코드 리팩토링 (기능 변화 없이 가독성이나 성능을 개선한 경우)
- **`test`** : 테스트 코드 추가 및 수정
- **`chore`** : 빌드 업무 수정, 패키지 매니저 설정, 환경 변수 수정 등

### 📝 실제 커밋 예시

- `feat: #3 - 카카오 로그인 구현 및 토큰 발급 기능 추가`
- `fix: #12 - 만료된 access_token이 제대로 지워지지 않던 에러 수정`
- `docs: #1 - 리드미 파일에 API 환경 변수 설정법 추가`

## 📌 4. Pull Request (PR) 및 Code Review 규칙

작업이 완료된 `feat/...` 브랜치를 `dev` 브랜치에 합치기 전, **반드시 서로의 코드를 리뷰하고 승인(Approve)받아야 합니다.**

### 🚨 PR 및 Merge 대상 규칙

1. 모든 `feat` 브랜치는 작업 완료 후 **`dev` 브랜치를 Base로 하여 PR을 올립니다.**
2. 자잘한 작업 커밋들이 커밋 히스토리를 더럽히지 않도록, `dev` 브랜치로 머지할 때는 GitHub에서 **`Squash and merge`** 방식을 기본으로 사용합니다.
3. `dev` 브랜치에서 충분한 기능 테스트와 전체 코드 조율이 완료된 안정적인 배포본만 **`dev ➡️ main`** 브랜치로 합치게 됩니다.

### ⚠️ 룰: 1인 이상 승인 필수

상대방 개발자의 Approve가 있어야만 `dev` 브랜치로 머지(Merge)할 수 있습니다.

### 📝 PR 작성 템플릿

Markdown

```
## 🔗 관련 이슈
- Close #이슈번호

## 📝 작업 내용
- [ ] 이번 PR에서 구현하거나 수정한 주요 핵심 내용을 적어주세요.

## 📸 스크린샷 (선택)
- UI 변경 사항이 있다면 스크린샷을 첨부해 주세요.

## 💬 리뷰어에게 부탁할 점
- 특별히 코드 리뷰를 꼼꼼히 받고 싶은 부분이나 고민했던 지점을 적어주세요.
```

### 🤝 코드 리뷰 가이드라인

- **피드백은 둥글게**: "이 코드는 틀렸어요"보다는 "이 방식으로 작성하면 성능/가독성 면에서 이런 장점이 있을 것 같은데 어떻게 생각하시나요?"와 같이 제안형으로 피드백을 주고받습니다.
- **빠른 피드백**: 알림이 오면 가급적 반나절 이내(늦어도 24시간 이내)에 코드를 확인하고 피드백을 남기거나 승인합니다.

## 🛠️ 5. 초기 환경 세팅 및 협업 룰

- **패키지 매니저 통일**: 우리 프로젝트는 패키지 설치 시 `npm`으로 완전히 통일하여 사용합니다. (Lock 파일 충돌 및 버전 꼬임 방지)
- **Node.js 버전 통일**: 로컬 개발 환경의 Node.js 사양은 **`v20 LTS`** 이상으로 일치시킵니다.
- **환경 변수 파일 (`.env`) 공유**: 로컬 환경 변수 파일은 보안 및 충돌 방지를 위해 절대 Git에 올리지 않으며(`gitignore` 등록 필수), 신규 환경 변수가 추가될 경우 노션의 환경 변수 관리 페이지 또는 메신저를 통해 상대 개발자에게 즉시 공유합니다.

### 🛠️ 세부 작업 순서 (매뉴얼)

#### 1단계: 깃허브에서 일거리 만들기 (Issue)

1. GitHub Repository의 **Issues** 탭으로 이동합니다.
2. **New Issue**를 누르고, 노션 가이드에 있는 템플릿을 복사해 내가 오늘 할 일(예: 카카오 로그인 연동)을 작성합니다.
3. 생성된 이슈의 번호(예: `#3`)를 기억해 둡니다.

#### 2단계: 내 컴퓨터(로컬) 코드 최신화하기

새 작업을 하기 전에는 항상 동료가 `dev`에 합쳐놓은 코드가 있을 수 있으므로 **로컬을 가장 최신 상태로 만들어야 합니다.**

1. `dev` 브랜치로 이동합니다:Bash
    
    ```
    git checkout dev
    ```
    
2. 최신 코드를 땡겨옵니다:Bash
    
    ```
    git pull origin dev
    ```
    

#### 3단계: 작업 브랜치(`feat`) 파기

최신 상태가 된 `dev` 브랜치 위에서 나만의 작업 영역을 새로 만듭니다.

1. 브랜치를 생성하고 바로 이동합니다 (이름 이니셜/닉네임과 이슈 번호 포함):Bash
    
    ```
    # 예시: feat/담당자-#이슈번호-간략설명
    git checkout -b feat/john-#3-kakao-login
    ```
    

#### 4단계: 폭풍 코딩 및 커밋 (`feat`)

열심히 코딩을 진행하면서 의미 있는 단위로 커밋을 쪼개어 기록합니다.

1. 수정한 파일을 스테이징 영역에 올립니다:Bash
    
    ```
    git add .
    ```
    
2. 규칙에 맞춰 커밋 메시지를 작성합니다:Bash
    
    ```
    # 예시: feat: #이슈번호 - 설명
    git commit -m "feat: #3 - 카카오 로그인 SDK 설치 및 초기화"
    ```
    

#### 5단계: 깃허브(원격)에 브랜치 올리기

드디어 내 컴퓨터에만 있던 브랜치와 코드를 깃허브에 처음으로 등장시킵니다.

1. 첫 push 때는 `u` 옵션을 붙여서 업스트림을 지정합니다:Bash
    
    ```
    git push -u origin feat/john-#3-kakao-login
    ```
    
    *(이후 추가적인 커밋을 올려서 push할 때는 그냥 `git push`만 쳐도 올라갑니다.)*
    

#### 6단계: Pull Request (PR) 작성 및 합치기 (Merge)

1. GitHub 웹페이지에 들어가면 상단에 노란색으로 **[Compare & pull request]** 버튼이 떠 있습니다. 클릭합니다.
2. **Base 브랜치가 `dev`로 설정되어 있는지 반드시 확인합니다.** (가장 중요! `main`이 아닙니다)
3. PR 템플릿 양식에 맞춰 내용을 작성하고 생성합니다.
4. 동료 개발자에게 *"나 #3번 기능 다 만들었으니 코드 리뷰 좀 부탁해!"*라고 슬랙이나 톡으로 알립니다.
5. 동료가 코드를 보고 피드백을 남기거나 Approve(승인)를 해주면, **`Squash and merge`** 버튼을 눌러 `dev` 브랜치에 코드를 최종 합칩니다.

#### 7단계: 뒤정리 및 다음 작업 준비

`dev` 브랜치에 완벽히 내 코드가 합쳐졌다면, 내 컴퓨터에 남아있는 작업 브랜치는 수명이 다한 것입니다.

1. 로컬의 중심 브랜치로 돌아갑니다:Bash
    
    ```
    git checkout dev
    ```
    
2. 방금 합쳐진 최신 버전의 `dev` 코드를 다시 가져옵니다:Bash
    
    ```
    git pull origin dev
    ```
    
3. 이제 다 쓴 내 로컬의 `feat` 브랜치는 깔끔하게 삭제합니다:Bash
    
    ```
    git branch -d feat/john-#3-kakao-login
    ```
    

### 💡 꿀팁 하나 더!

작업 중에 동료 개발자가 `dev` 브랜치에 먼저 합친 코드(예: 공통 API 세팅 등)를 내 작업 브랜치(`feat/...`)로 급하게 가져와서 이어서 개발해야 할 때가 있습니다.

그럴 때는 내 작업 브랜치에서 아래 명령어를 실행해 주면 됩니다:

Bash

```
# 내 작업 브랜치 상태에서 실행
git pull origin dev
```

이렇게 하면 동료가 짜둔 최신 코드가 내 작업 영역으로 쏙 들어와서 충돌을 미리 방지하며 개발할 수 있습니다.
